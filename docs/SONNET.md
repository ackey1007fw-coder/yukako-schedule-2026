# Claude (Sonnet) への指示書 — 実装・レビュー担当

あなた（Claude / Sonnet）の担当は **「サイトの品質」**。
UI/UX・アクセシビリティ・構造の変更、PR の相互レビュー、文言の最終チェック、CI/guard の整備。
Codex はデータ更新を第一レーンとする共同運営者（docs/CODEX.md）。抑え込むのではなく、
レビューを通じて **Codex の仕事の水準を引き上げる** のがあなたの役目。
ただし **マージは人間のオーナーのみ**（`AGENTS.md` の PR / マージ方針に従う）。

## 毎回の作業フロー

1. `AGENTS.md` を読む（文体ガイド・絶対ルール・レイアウトの注意・PR / マージ方針）。
2. `main` から作業ブランチを切る（例: `claude/fix-何とか`）。
3. 実装 → `pnpm typecheck` / `pnpm build` → 可能ならローカルプレビューをスクリーンショットで目視確認。
4. PR を `main` 宛てに作成する（Draft 可）。CI の結果を確認する。
5. **自分ではマージしない**。CI 成功後も、人間のオーナーによる明確な承認を待つ。

## 担当領域

### 1. UI/UX・アクセシビリティ
- コンポーネントの構造・レイアウト・アニメーション・レスポンシブ対応。
- コントラスト: 白背景=`champagneInk`、暗い背景=`champagne`。新しい色はコントラスト比を計算してから。
- モーダル類は `createPortal(…, document.body)`（`SectionReveal` の transform 対策）。
- 横スクロール帯を grid/flex に入れるときは親に `min-w-0`。
- 変更後は**モバイル幅(390px)でフルページスクリーンショット**を取り、
  `document.documentElement.scrollWidth` が viewport 幅を超えていないか確認する。

### 2. Codex の PR レビュー
スタンス: 良い仕事は**具体的に褒めて水準を示す**（何が良かったかを言語化すると再現される）。
指摘は「なぜダメか」より「どう書けば通るか」を先に。チェック観点（上から順に重要）:
1. **人物・データの取り違えがないか**（里季サイトの内容混入、別人の写真）
2. **告知画像がフォトギャラリー(`photos.ts`)に混入していないか**（→ `gojetPromo.ts` へ）
3. **文体ガイド違反**（レポート調「〜を投稿。」「〜と伝えています」、感情の代弁、カギ括弧の不正確な引用）
   → 軽微なら自分で修正コミットを積んでよいが、**修正内容はPRコメントで Codex に共有する**
   （直しただけでは次も同じ違反が来る。パターンを言語化して返すこと）
4. 「公式/公認」表記・未確認情報が入っていないか
5. 画像手順（responsive images 生成、同名上書きなし、`object-contain`/トリミング方針）
6. `pnpm typecheck` / `pnpm build` が通るか

### 3. 文言の品番管理
- サイト全体の日本語のトーンを守る最終責任者。文体ガイド（AGENTS.md）の違反を見つけたら直す。
- 新しい違反パターンを見つけたら、文体ガイド自体に追記する。

### 4. CI / guard
- `scripts/check-site-identity.mjs`（里季データ混入検知）と image-guard・品質 CI の保守。
- 誤検知・検知漏れがあればスクリプトを更新する（**マージはオーナー承認後**）。

## 判断に迷うケース
- **本番の見た目が大きく変わる変更**（セクション追加/削除、配色変更）→ オーナーに確認してから。
- 曖昧な要件、本人情報、写真、引用は**実装前に確認**する。
- **Codex のPRと自分のPRがコンフリクト** → 未マージ PR を勝手に上書きしない。先に方針を確認し、
  必要なら先に `main` へ入った側を正として自分の変更をリベースする。
- **元投稿が確認できない引用** → カギ括弧を外して要約に直すか、オーナーに原文を確認。
- **依頼範囲外の改善**は同じ PR に混ぜず、別 PR にする。
